## 🚀 AI Helper 사용 가이드

### 1단계: API 키 설정
프로젝트 루트 폴더에 생성된 `.env` 파일을 메모장이나 에디터로 열어 사용 중이신 OpenRouter API Key를 입력합니다.
```env
OPENROUTER_API_KEY=sk-or-v1-본인의_API_키_입력
```

### 2단계: 서버 재실행
```powershell
python server.py
```
서버를 시작하고 웹 브라우저에서 접속합니다.

### 3단계: AI 분석 실행 및 승인
1. DXF 도면을 업로드하고 기존 규칙 기반 1차 분석을 완료합니다.
2. 우측 상단 탭에서 **기둥** 또는 **보** 검증 화면으로 이동합니다.
3. 리스트 헤더 영역에 추가된 **`[🤖 AI 기둥 분석]`** 혹은 **`[🤖 AI 보 분석]`** 버튼을 클릭합니다.
4. AI 분석이 수행된 후 오른쪽 사이드에 보라색 **AI 분석 추천 결과** 패널이 슬라이드인됩니다. 동시에 도면 위에 보라색으로 감지된 AI 기둥/보가 마킹됩니다.
5. 추천 목록에서 판정 근거와 도면 위치를 대조하고, **[승인]**을 누르면 정식 기둥/보 리스트에 추가되어 즉시 최종 적산 중량 집계표와 보고서에 포함됩니다. (승인 부재는 최종 출력물 비고 란에 `"AI 제안 승인"` 표기)

### 4단계: AI 높이 추천 기능 실행
1. 기둥 리스트 중 높이를 설정하려는 기둥을 선택합니다.
2. 높이 직접 입력창 우측 상단에 위치한 **`[🤖 AI 높이 추천]`** 버튼을 클릭합니다.
3. AI가 도면(또는 선택한 참조 도면)을 분석하여 가장 긴 높이 수치를 계산하고 근거와 함께 팝업 대화창으로 제시합니다.
4. **[확인(적용)]**을 누르면 기둥 높이 입력 창에 해당 수치(예: 4200mm)가 자동 기입됩니다.

## 🛠️ 기둥 높이 AI 추천 기능 개선 (수직선 기준 높이 필터링 및 수평선 길이 제외)

기존에 **"AI 높이 추천"** 기능 사용 시, 도면의 수직선 기준 높이(층고)를 구해야 하는데 지붕의 수평선 길이(예: 10,000mm, 5,000mm 등)를 AI가 추천하는 오작동 문제를 해결하였습니다.

### 🔍 원인 분석 (Root Cause)
1. **DIMENSION 엔티티 파싱 중 예외 발생 및 스킵**:
   `takeoff_analysis.py`의 `extract_height_texts_for_ai`에서 `text_override` 속성을 읽어오려다 ezdxf의 `Invalid DXF attribute` 에러가 발생했습니다. 이로 인해 치수선(`DIMENSION`) 엔티티를 파싱하여 수직 방향을 판단하는 로직이 전부 스킵(continue)되었습니다.
2. **도면 텍스트의 방향성 정보 유실**:
   백엔드에서 `global_texts`를 모을 때 및 프론트엔드(`index.html`)에서 수집된 텍스트 목록을 백엔드 API에 넘길 때, 개별 텍스트 및 치수선의 수직/수평 각도 정보(`direction`)를 전송하지 않아 LLM이 단순 수치 크기만 보고 지붕의 수평 폭(10000 등)을 가장 긴 층고로 판단하게 되었습니다.

### 💡 개선 내용 (Fixes Applied)
1. **DIMENSION 엔티티 파싱 에러 수정 및 방향 판별 구현 (`takeoff_analysis.py`)**:
   - `text_override` 속성 대신 ezdxf 표준 속성인 `text`를 사용하여 에러 발생을 해결했습니다.
   - 치수선의 정의점(`defpoint`, `defpoint2`, `defpoint3`)의 좌표 차이(`dx`, `dy`)를 비교하여 **수직 치수선(`direction: 'vertical'`)**과 **수평 치수선(`direction: 'horizontal'`)**을 정확하게 판별하도록 로직을 강화했습니다.
2. **수평 치수선 원천 배제 필터링 (`server.py` & `takeoff_analysis.py`)**:
   - **백엔드 직접 추출 방식**: `extract_height_texts_for_ai` 함수가 동작할 때 수평선 방향(`direction: 'horizontal'`)의 치수는 높이 추천 후보군에 수집되지 않도록 원천 필터링했습니다.
   - **프론트엔드 연동 방식**: 프론트엔드(`index.html`)에서 텍스트 좌표 목록을 넘길 때 `direction` 필드를 추가로 넘기도록 바인딩을 추가하고, 백엔드 API(`server.py`) 수신 측에서 `direction == 'horizontal'`인 데이터는 기둥 높이 분석 후보군에서 완전히 걸러내어 LLM에 아예 제공되지 않도록 수정했습니다.

### 🧪 검증 결과 (Verification Results)
- 검증 스크립트(`scratch/test_ai_height.py`)를 통해 `도면2.dxf`를 분석해 본 결과:
  - 단면도 영역 내 지붕 수평 폭(10000mm, 5000mm 등)을 포함한 **수평선 길이 치수(`Horizontals: 0`)는 전량 추천 후보에서 필터링**되었습니다.
  - 기둥의 실제 높이인 **수직선 기준 높이 치수(`6000`, `7700`, `4000` 등)만 정확하게 추출 및 AI에 전달**됨을 확인했습니다.
3. **실시간 치수선 방향 검증 및 필터링 기능 강화 (`server.py` API)**:
   - 사용자가 도면을 재분석하지 않았거나 구버전 캐시 데이터를 지닌 브라우저 세션에서 호출하는 경우, 데이터 내 `direction` 필드가 누락되어 `"unknown"`으로 백엔드에 전송될 수 있습니다.
   - 이를 방지하기 위해 백엔드 API인 `/api/ai_recommend_height` 단에서 **실시간 크로스 체크(Cross-reference Validation)**를 추가하였습니다.
   - 백엔드의 검증된 수직 치수선 추출기(`extract_height_texts_for_ai`)에서 얻은 수직 치수선들의 좌표 목록과 프론트엔드가 보낸 `type == 'dimension'` 데이터를 실시간으로 좌표 비교 및 검증합니다.
   - 좌표가 매칭되지 않는 치수선(즉 수평선인 지붕의 10000 등)은 백엔드단에서 강제적으로 필터링 및 제외하여 오작동 가능성을 철저하게 0%로 차단했습니다.

### 📐 5. PDF 도면 내 AI 제안 동그라미 점선 마커 유실 오류 수정 ([index.html](file:///d:/python_2025-26/aiffel_p/c_partner/final/index.html))
PDF의 "산출 근거 도면" 및 "기둥 높이 산출 근거 도면"에서 AI 승인/매칭 결과물(동그라미 점선 원)이 정상적으로 렌더링되지 않던 논리적 버그를 해결하였습니다.
1. **AI 추천 높이 적용 시 `source: 'both'` 마킹 연동**:
   - 기존에는 AI 높이 추천을 적용하더라도 기둥의 높이(`height`) 필드만 갱신되고, 이 정보가 AI를 통해 취득되었다는 속성(`source: 'both'`)이 기둥 데이터에 누락되어 있었습니다.
   - `window._isAiHeightApplied` 상태 플래그를 도입하여, AI 추천 높이를 입력란에 주입한 후 [적용] 또는 [일괄 적용]을 누르는 시점에 해당 기둥의 `source` 속성을 `'both'`로 정상 매칭해 주도록 수정했습니다.
2. **참조 시트 기둥/보 데이터 PDF 전송 누락 해결**:
   - 기존에는 사용자가 메인 시트만 검증 완료 승인하고 참조 시트는 따로 승인하지 않았을 경우, PDF 전송용 페이로드(`runFinalExcelBuild`)를 빌드할 때 참조 시트의 기둥/보가 완전히 필터링(유실)되었습니다.
   - 이로 인해 "기둥 높이 산출 근거 도면"을 그릴 때 필요한 기둥 정보 자체가 백엔드로 넘어가지 않았습니다.
   - 메인 시트의 승인 여부뿐만 아니라, 메인 시트가 참조하고 있는 **참조 시트 ID(`height_ref_sheet_id`)**도 전송용 시트 필터 셋에 추가로 취합하도록 수정하여 모든 데이터가 누락 없이 PDF 렌더러로 정상 전달되도록 수정했습니다.

### ⚡ 7. 대용량 도면(도면3 등) AI 분석 시 토큰 폭발 및 응답 잘림(Unterminated String) 현상 해결 ([takeoff_analysis.py](file:///d:/python_2025-26/aiffel_p/c_partner/final/takeoff_analysis.py#L3016))
도면3(52.9MB)과 같이 도면 내에 엄청난 수의 텍스트가 존재하는 경우, AI 컨텍스트로 전달되는 텍스트 양이 수천 개에 달해 OpenRouter/Gemini API의 출력 한계값을 초과하여 JSON 응답 도중 문자열이 잘리는 `Unterminated string` 디코딩 실패 오류가 발생하는 문제를 해결하였습니다.
1. **AI 텍스트 전처리 필터링 추가 (`is_valuable_ai_text`)**:
   - 도면 내에 존재하는 다량의 한글 주석(예: "제물CONC마감", "철근콘크리트", "도면틀" 등) 및 너무 길거나 특수기호로만 된 텍스트들을 AI 분석 전송 후보군에서 100% 필터링하여 제외하도록 설계하였습니다.
   - 보/기둥 부호 정규식에 부합하거나, 기하 규격(예: `H-250*125`), 단순 숫자(치수 값) 등 AI가 분석에 반드시 필요로 하는 유효 텍스트만 추출하도록 화이트리스트 조건을 강화했습니다.
2. **토큰 절감 및 모델 응답 성공률 극대화**:
   - 이 정제를 통해 대용량 도면의 텍스트 전송량이 기존 대비 90% 이상 절감되어, 입력 토큰 낭비를 막고 출력 토큰 한계로 인한 JSON 디코딩 에러를 완벽하게 차단하였습니다.
